# 代码审查强制清单 v1.2

> **使用说明**：本清单提炼自四个工业级项目的真实 Bug 和架构缺陷。AI 生成代码后，必须让 AI 逐项对照本清单进行自审。每项规则标注了来源项目，便于追溯。
>
> **如何使用**：将本清单全文 + 附录中的审查提示词一起发给 AI。
>
> **N/A 处理规则**：若某个类别与当前项目技术栈完全无关（如非 AI 项目遇到第三节"AI Provider 抽象"），在审查报告中标注 `N/A — [原因]` 并跳过，无需逐项检查。

## 一、单例与生命周期管理

| # | 检查项 | 错误示例 | 正确做法 | 来源 |
|---|--------|---------|---------|------|
| 1.1 | **有状态的对象是否被注册为单例？** | `@Bean public KieSession kieSession()` 返回非线程安全的 KieSession | KieContainer 单例，KieSession 每次请求新建并用后 dispose | 医保系统 |
| 1.2 | **单例对象是否在运行时被意外销毁？** | `session.dispose()` 在 finally 中销毁了全局单例 | 单例对象不应提供 dispose/destroy，或确保调用者无权销毁 | 医保系统 |
| 1.3 | **Provider/Client 是否作为单例正确复用？** | 每次请求 `new Anthropic()` 导致连接池失效 | 使用单例模式或依赖注入容器管理生命周期 | Seed AI |
| 1.4 | **数据库连接池是否配置了合理的最大连接数？** | 使用默认配置，并发时连接耗尽 | 根据并发量设置 `maximumPoolSize`，并配置连接超时 | 三个项目均有 |
| 1.5 | **【Python/FastAPI】async Session 是否作为模块级全局变量？** | `async_session = AsyncSession(engine)` 声明在模块顶层，多请求共享同一 Session | 必须使用 `async_sessionmaker` 工厂 + FastAPI `Depends(get_session)` 依赖注入，每请求独立 Session | TaskFlow API |

## 二、缓存设计

| # | 检查项 | 错误示例 | 正确做法 | 来源 |
|---|--------|---------|---------|------|
| 2.1 | **业务决策结果是否被缓存？** | `@Cacheable` 注解在审核结果方法上，规则变更后旧结论仍生效 | 只缓存元数据/规则定义，不缓存动态决策结果 | 医保系统 |
| 2.2 | **缓存是否有明确的失效策略？** | 缓存永不过期，数据持续陈旧 | 必须设置 TTL（如 web_fetch 5分钟），或在写操作时主动失效 | Seed AI |
| 2.3 | **写操作前是否先失效了相关缓存？** | `file_write` 执行前未清除 `file_read` 缓存，导致脏读 | 写操作执行前，按路径或模式批量失效相关缓存 | Seed AI |
| 2.4 | **缓存 Key 是否包含所有影响结果的变量？** | 仅用文件名做 Key，忽略行范围参数 | Key 应为 `toolName + JSON.stringify(全部入参)` | Seed AI |
| 2.5 | **缓存是否可能导致敏感数据泄露？** | 用户A的审核结果被用户B命中 | 缓存 Key 必须包含用户/租户隔离标识 | 通用原则 |

## 四、错误处理与日志

| # | 检查项 | 错误示例 | 正确做法 | 来源 |
|---|--------|---------|---------|------|
| 4.1 | **工具层是否向外抛出异常？** | `throw new Error("file not found")` 导致 Agent Loop 中断 | 工具层 catch 所有异常，返回 `{ isError: true, content: "错误描述" }` | Seed AI |
| 4.2 | **catch 块是否为空？** | `try { ... } catch (e) {}` 静默吞掉错误 | 至少记录日志 `logger.warn("operation failed", e)` | 三个项目均有 |
| 4.3 | **非致命错误是否导致整个流程中断？** | 记忆加载失败 → 会话启动失败 | 非致命模块（记忆、沙箱、压缩）失败时降级而非崩溃 | Seed AI |
| 4.4 | **结构化日志是否包含足够上下文？** | `log.error("failed")` 无任何参数 | 必须包含 `operation`、`error`、`userId`、`traceId` 等字段 | 三个项目均有 |
| 4.5 | **是否有全局异常处理器？** | Controller 层未 catch 的异常直接暴露给客户端 | 使用 `@ControllerAdvice` 统一处理，返回标准错误格式 | 医保系统 |

## 五、依赖管理与编译

| # | 检查项 | 错误示例 | 正确做法 | 来源 |
|---|--------|---------|---------|------|
| 5.1 | **go.mod / package.json / pom.xml 是否包含所有实际使用的依赖？** | 代码中 import 了包但未在依赖文件中声明 | `go mod tidy` 或 `npm install --save` 确保一致 | CDN 系统 |
| 5.2 | **是否使用了正确的包版本？** | Spring Boot 3.x 项目仍使用 `javax.validation` | 检查 Jakarta 迁移，所有 `javax.*` 改为 `jakarta.*` | 医保系统 |
| 5.3 | **import 语句位置是否正确？** | `import "context"` 写在文件末尾（Go 语法错误） | import 必须在文件顶部 import 块内，运行 linter 自动检查 | CDN 系统 |
| 5.4 | **多个文件是否定义了同名函数/变量？** | `database.go` 和 `migrations.go` 都定义了 `RunMigrations`，编译冲突 | 删除冗余实现，保留唯一权威源 | CDN 系统 |
| 5.5 | **是否遗漏了必要的框架注解？** | MyBatis Plus 枚举字段未加 `@EnumValue`，按字符串存储导致类型不匹配 | 枚举字段必须用 `@EnumValue` 标注存储值 | 医保系统 |
| 5.6 | **Service 接口是否有对应的实现类？** | `RuleService` 接口无 Impl，启动即 `NoSuchBeanDefinitionException` | 每个 Service 接口必须有一个 `@Service` 实现类 | 医保系统 |

| # | 检查项 | 错误示例 | 正确做法 | 来源 |
|---|--------|---------|---------|------|
| 6.1 | **是否有硬编码的密钥/密码？** | `apiKey = "test-api-key-123456"` 硬编码在 middleware.go | 从环境变量读取，开发默认值只用于本地 | CDN 系统 |

## 附录：如何使用本清单

### 在阶段二（生成代码前）注入约束

将清单的第一、六节作为约束前缀发给 AI，预防最高频的问题：

```
在生成代码之前，请确认你已理解并将遵守以下强制约束：
[粘贴：第一节（单例）+ 第六节（安全）的全部检查项]
```

### 在阶段三（代码审查时）逐项检查

将本清单全文发给 AI，附加指令：

```
请对照《代码审查强制清单》，逐项检查当前项目的所有源代码。
对每一项，回答 ✅ 符合 / ❌ 违反 / ⚠️ 部分符合。
对于 ❌ 违反的项，必须指出具体文件、行号和修复建议。
最终生成审查报告，按严重程度（P0/P1/P2）分级。
```

### 在阶段六（知识沉淀时）更新本清单

每完成一个项目，回顾是否有新的 Bug 模式值得加入。如有，追加到对应章节并标注来源项目。
