# 贡献指南

感谢你对本方法论库的贡献。本库的核心价值来自真实项目的经验积累，任何形式的贡献都很欢迎。

---

## 贡献方式

### 1. 归档新案例（最高价值）

如果你用本库的工作流完成了一个项目，可以将项目经验归档为案例：

```
1. 将项目的 /memory/项目操作记忆.md 复制到 case-studies/
2. 命名格式：[项目名]-[YYYY-MM].md
3. 在 case-studies/README.md 的案例列表中追加一行
4. 如发现新的高频踩坑，追加到 case-studies/README.md 的 TOP 10 表格
```

### 2. 改进模板（摩擦点修复）

如果在使用过程中发现模板有不清晰、缺失或不适用的地方：

```
1. 在 Issue 中描述：[场景] 什么情况下 → [问题] 遇到什么摩擦 → [建议] 应该如何修改
2. 或直接提交 PR，在修改说明中注明摩擦点来源（哪个项目/阶段触发）
```

### 3. 新增语言约束（附录 B 扩展）

`templates/workflow-template.md` 附录 B 目前覆盖 Java / Go / TypeScript / Python。  
如果你有 Rust / C++ / Kotlin 等语言的高频 AI 错误模式，欢迎补充：

```
格式参考现有语言约束块，每条约束附简短的"根因"说明
```

### 4. 追加清单规则（发现新 Bug 模式）

`checklists/code-review-checklist.md` 每次新项目都可能发现新 Bug 模式：

```
1. 确认该 Bug 模式有足够的普遍性（不只是一次偶发错误）
2. 加入对应的章节（按类型分类）
3. 在 PR 说明中注明：从哪个项目 / 哪次 AI 生成中发现
```

---

## 提交规范

PR 标题格式：

```
[类型] 简短描述

类型：
- case  — 新增案例存档
- fix   — 修复模板错误或无效内容
- feat  — 新增清单规则 / 约束条目 / 附录内容
- docs  — 说明文档改进
```

---

## 不接受的贡献

- 与 AI 协作工作流无关的内容
- 未经实际项目验证的纯理论内容
- 大规模重构现有结构（请先开 Issue 讨论）

---

*如有任何问题，欢迎通过 Issue 沟通。*
