# 代码审查提示词

> **用途**：阶段三（系统性代码审查）使用。将本文件内容 + `code-review-checklist.md` 全文一起发给 AI。  
> **触发时机**：AI 完成初始代码生成、编译验证通过后。  
> **预期产出**：结构化的《审查报告》，含 P0/P1/P2 分级问题列表。

---

## 使用方法

1. 打开 `E:\AI-Methodology\checklists\code-review-checklist.md`，复制全文
2. 在新的对话中，先粘贴 **清单全文**，再粘贴下面的 **审查指令**
3. 等待 AI 输出审查报告后，人工确认优先级，再进入阶段四

---

## 审查指令（直接粘贴给 AI）

```
请对当前项目的全部源代码执行系统性审查。

【审查范围】
- 所有 /src 目录下的源代码文件
- 依赖配置文件（pom.xml / package.json / go.mod）
- Dockerfile 和 docker-compose.yml
- 测试文件（/tests 目录）

【审查步骤】
1. 阅读上方《代码审查强制清单》的全部 10 个类别（约 45 个检查项）
2. 逐一检查当前项目代码，对每项标注状态：
   - ✅ 符合
   - ❌ 违反（必须给出：文件名 + 行号 + 问题描述 + 具体修复代码）
   - ⚠️ 部分符合（说明差距和风险等级）
   - N/A 不适用（说明原因）

【输出格式】

---《审查报告》---

## P0 致命缺陷（系统崩溃 / 安全漏洞 / 无法编译）

| # | 问题描述 | 文件:行号 | 修复方案（代码级） |
|---|---------|---------|----------------|
| 1 | [清单编号] 问题描述 | src/xxx.java:42 | 修复代码或步骤 |

## P1 架构缺失（功能不完整 / 核心模块缺失）

| # | 问题描述 | 文件:行号 | 修复方案 |
|---|---------|---------|---------|
| 1 | | | |

## P2 代码异味（不规范但当前可工作）

| # | 问题描述 | 文件:行号 | 建议 |
|---|---------|---------|-----|
| 1 | | | |

## 通过项 ✅
（列出所有符合的检查项编号，如：1.1 ✅ 1.2 ✅ 2.3 ✅ ...）

## N/A 项
（列出不适用的项目及原因）

## 审查统计与建议

| 指标 | 数值 |
|------|------|
| P0 数量 | X 个 |
| P1 数量 | X 个 |
| P2 数量 | X 个 |
| 建议修复轮次 | X 轮 |
| 预计第几轮后可运行 | 第 X 轮结束后 |
| 最高风险项 | [一句话描述] |

---《报告结束》---

【重要】
审查完成后，请等待我确认优先级。不要立即开始修复。
每个 P0 问题必须包含可以直接执行的修复代码，不允许只说"应该修复"。
```

## 快速审查模式（仅审查最高危区域）

当时间有限时，可以只审查以下 4 个最高频问题区域：

```
请快速审查当前项目中最容易出问题的 4 个区域：

1. 【单例与生命周期】检查所有 @Bean / 单例注入的对象，是否有有状态对象被错误地作为单例
2. 【硬编码密钥】在所有源文件中搜索 password、secret、apiKey、token 等关键词，检查是否有硬编码
3. 【缺失实现】检查所有 Service 接口，是否每个都有对应的 @Service 实现类
4. 【编译安全性】检查 import 语句是否在文件顶部，是否有重复定义的函数/变量

对每个问题区域，输出：
- 是否有问题（✅ 无 / ❌ 有）
- 如果有问题，给出文件名 + 行号 + 一句话描述
```
