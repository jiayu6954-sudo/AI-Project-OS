## 一、项目定义书

**系统定位**：基于国家医保局智能监管改革试点要求，构建事前（开单预警）、事中（住院动态监控）、事后（疑点筛查）全流程闭环监管平台。

**项目背景**：试点政策驱动，医院 HIS 系统产生的数据需要实时审核，防止不合规医疗行为（超量用药、违禁药品、过度检查等）。审核结果需持久化以支持事后合规审计。

### 硬性约束

| 类别 | 约束内容 | 量化指标 |
|------|---------|---------|
| 性能 | 实时审核响应时间 | P95 ≤ 300ms，P99 ≤ 800ms |
| 规则 | 本地化规则数量 | 200+ 条，支持热更新（不重启服务） |
| 安全合规 | 数据安全等级 | 等保三级，患者数据脱敏存储 |
| 可用性 | 系统可用性 | ≥ 99.9% |

### 成功标准

- [ ] 8 个微服务全部完整实现（gateway/audit/rule/knowledge/data/user/report/ai-service）
- [ ] 20 条 DRL 规则正确执行（drug-rules 10条 + treatment-cost-rules 10条）
- [ ] 核心场景端到端通过（他达拉非禁女性 → BLOCK 级别）
- [ ] CI 流水线绿色，Docker 镜像可在干净环境运行
- [ ] 文档完整，可供机构评估

### 技术栈决策

| 组件 | 最终选型 | 核心原因 | 主要备选 |
|------|---------|---------|---------|
| 语言/框架 | Java 17 + Spring Boot 3.x | 企业级标准，团队熟悉 | Spring Boot 2.x（已不再维护） |
| 规则引擎 | Drools 8.x | 业界标准，支持热更新、复杂规则逻辑 | 自研规则表达式 |
| ORM | MyBatis Plus | 简洁，SQL 可控，分页插件完善 | JPA/Hibernate |

| 缓存 | Redis DB0-4 | 会话/规则/知识库分库管理 | Memcached |
| 服务发现 | Nacos | 国产主流，与 Spring Cloud Alibaba 集成好 | Consul |
| AI 服务 | Python FastAPI + IsolationForest | 异常检测 + NLP，Python 生态更成熟 | Java ML 库 |

- 不做移动端

## 二、关键设计决策

| # | 决策点 | 最终选择 | 放弃的方案 | 决策理由 |
|---|--------|---------|-----------|---------|
| D-1 | KieSession 生命周期 | 每次请求新建，用后立即 dispose | Spring Bean 单例 + 会话池 | 医疗审核场景正确性 > 性能；有状态会话不能共享，状态污染会导致串单 |
| D-2 | DRL 结果收集方式 | `global List auditDetails; auditDetails.add(detail)` | Insert 返回对象 | Global List 在并发规则执行中更稳定，是 Drools 官方最佳实践 |
| D-3 | JWT 验证位置 | 网关统一验证，解析后注入请求头 `X-User-Id` / `X-User-Roles` | 各微服务自验证 | 下游服务无需引入 JWT 依赖，逻辑集中，白名单配置唯一 |
| D-4 | 审核结果存储结构 | 独立列（`passed BOOLEAN`, `highest_level INTEGER`） | `audit_result JSONB` | 报表服务需要 `GROUP BY highest_level` 聚合查询，独立列性能远优于 JSONB 提取 |
| D-5 | 测试分层策略 | 单元测试（Mockito）+ Controller切片（@WebMvcTest）+ 集成测试（Testcontainers） | 全部集成测试 | 规则引擎测试必须用真实 Drools，但业务逻辑可以 Mock；分层避免 CI 依赖外部服务 |

## 三、审查报告（初始代码审查结果）

### P0 致命缺陷（6 个）

| # | 问题描述 | 文件:位置 | 修复方案 |
|---|---------|---------|---------|
| 1 | KieSession 注册为 Spring Bean 单例，第一次请求后 dispose 永久销毁 | `DroolsConfig.java` + `RuleEngineService.java` | KieContainer 单例，KieSession 每请求新建+finally dispose |
| 2 | `@Cacheable` 加在 `executeRules()` 上，规则变更后旧结论持续生效（医疗安全隐患） | `RuleEngineService.java` | 彻底移除 @Cacheable，规则执行结果不应被缓存 |
| 3 | `RuleService` 接口有声明无实现类，启动即 `NoSuchBeanDefinitionException` | `RuleService.java` | 创建 `RuleServiceImpl` + `RuleMapper`，完整实现13个接口 |
| 4 | 使用 `javax.validation` 而非 `jakarta.validation`（Spring Boot 3.x 已迁移到 Jakarta EE） | `AuditRequest.java`, `RuleDTO.java`, `RuleController.java` | 全局替换 javax.→jakarta. |
| 5 | Mock 对象无 `when()` 配置，Mockito 对 List 返回空集合，规则断言必然失败 | 测试文件 | 为每个 Mock 调用路径补全 when().thenReturn() |
| 6 | 使用 Java `assert` 而非 JUnit `Assertions`，JVM 默认不开启 assert，断言永不执行 | 测试文件 | 改用 `Assertions.assertTrue(duration < 1000, "超时")` |

### P1 架构缺失（8 个）

| # | 缺失项 | 影响 |
|---|--------|------|
| 1 | 无 DRL 规则文件（只有3条内嵌规则，1条空实现） | 核心业务无法运行 |
| 2 | 审核结果无持久化 | 无法事后分析、合规审计 |
| 3 | gateway/user/knowledge/data/report 五个服务为空目录 | 系统功能严重不完整 |
| 4 | 无 JWT 安全层 | 所有接口完全开放 |
| 5 | 无 Dockerfile | 无法容器化部署 |
| 6 | 无 CI/CD 流水线 | 无法持续交付 |
| 7 | `@MapperScan` 缺失 | MyBatis 数据库操作全部失败 |
| 8 | 集成测试依赖完整基础设施 | 无法在 CI 中运行 |

## 四、修复记录

### 第一轮（P0 修复）

**目标**：让系统能跑起来  
**主要修复**：KieSession 线程安全重构、移除危险 @Cacheable、实现 RuleService、javax→jakarta 迁移、完全重写测试  
**关键决策**：KieSession per-request 模式，医疗场景正确性第一

### 第二轮（P1 核心业务）

**目标**：让功能完整  
**主要修复**：创建 20 条 DRL 医保规则（drug-rules + treatment-cost-rules）、实现审核结果持久化、添加 @MapperScan + MetaObjectHandler

### 第三轮（P1 架构补全）

**目标**：让架构完整  
**主要修复**：Gateway（JWT鉴权+路由+限流+CORS）、User Service（登录+JWT+BCrypt+黑名单）、Knowledge/Data/Report 服务骨架、公共异常体系 BusinessException

### 第四轮（工程化）

**目标**：让系统可交付  
**主要修复**：全服务 Dockerfile（多阶段+非root）、docker-compose 升级（15容器+healthcheck）、deploy.sh 一键部署、.gitlab-ci.yml（6阶段流水线）

### 第五轮（100% 闭环）

**目标**：让系统生产就绪  
**主要修复**：DB Schema 与实体对齐（AuditRecord passed/highest_level）、枚举 @EnumValue、Knowledge/Report/Data 服务完整实现、Token Refresh、AI 服务（Python FastAPI + IsolationForest）、Testcontainers 集成测试  
**关键决策**：JSONB 改为独立列，因为报表服务需要聚合查询

### 第六轮（专业交付包）

**目标**：面向机构评估  
**主要修复**：医保智能审核系统白皮书（1300+行15章）、HIS 模拟器（4厂商7场景）、前端演示 Demo（离线可用）、生产安全配置（env模板+Nginx HTTPS+加密工具）

## 七、经验教训

1. **Spring Boot 3.x Jakarta 迁移**：AI 生成 SB3 代码时几乎必用 `javax.*`，必须在约束提示词中明确要求使用 `jakarta.*`
2. **Drools 生命周期铁律**：KieSession 绝不能作为 Spring Bean；有状态 + 单例 = 第一次请求后系统永久崩溃
3. **规则引擎缓存边界**：只缓存"规则定义/KieBase"，绝不缓存"规则执行结果"；医保规则变更可能在分钟级生效
4. **数据库设计要考虑下游查询**：JSONB 灵活但不支持 SQL 聚合，报表服务的 `GROUP BY / COUNT WHERE` 需要独立列
5. **MyBatis Plus + SB3 组合陷阱**：必须使用 `mybatis-plus-spring-boot3-starter`，不是 `starter`
6. **测试分层价值**：规则逻辑必须用真实 Drools + H2 集成测试，单纯 Mock 无法验证 DRL 规则正确性
