## 一、项目定义书

**系统定位**：一个 TypeScript CLI AI 编程助手，系统性研究 Claude Code 架构后从零实现，并在多 Provider 支持、跨会话记忆、本地 LLM、Docker 沙箱、研究子 Agent 等方向进行了 27 项针对性创新。

**项目背景**：Claude Code 只支持 Anthropic API，无跨会话记忆，无本地 LLM 支持。Seed AI 在研究其架构和 DeerFlow-2.0 后，补充了这些 Claude Code 刻意不覆盖的方向。定位为个人研究工程项目，非组织级产品。

### 硬性约束

| 类别 | 约束内容 | 说明 |
|------|---------|------|
| 多 Provider | 8 个 AI Provider | Anthropic / OpenAI / DeepSeek / Groq / Gemini / Ollama / OpenRouter / Moonshot |
| 本地 LLM | 自动发现 + 降级 | 检测 tool-call 能力，不支持时 XML fallback |
| 跨会话记忆 | 3 层记忆 + 语义检索 | user.md / context.md / decisions.md + 向量相似度检索 |
| 上下文管理 | 80% 窗口触发压缩 | Haiku 语义摘要，防止上下文溢出 |
| 迭代速度 | 快速发布 | 42 次迭代，alpha.25 |

## 二、关键设计决策

| # | 决策点 | 最终选择 | 放弃的方案 | 决策理由 |
|---|--------|---------|-----------|---------|
| D-1 | AI Provider 抽象 | `AIProvider` 接口 + 工厂方法 `createProvider(settings)` | 直接 `new Anthropic()` | 多 Provider 支持的核心前提；违反此原则3次才彻底稳定 |
| D-2 | 工具结果缓存 | 幂等读操作缓存（file_read/glob/grep/web_fetch），写操作执行前失效 | 不缓存 / 全量缓存 | 减少重复文件读取；写前失效防止脏读（如 file_write 后 file_read 返回旧内容） |
| D-3 | 上下文压缩触发点 | 80% 上下文窗口，Haiku 语义压缩，摘要注入系统提示 | 固定轮次截断 | 语义压缩保留关键决策；多轮累积摘要防止信息丢失 |
| D-4 | TUI 渲染策略 | Static（已完成消息写入终端滚动区，不重绘）+ Dynamic（只重绘流式区域，80ms） | 全量重绘 | 消除流式输出抖动；大型输出不会导致终端闪烁 |
| D-5 | 研究子 Agent | `spawn_research` 启动 isolated sub-loop，只有 web_search + web_fetch 工具 | 主 Loop 内执行 | 隔离研究上下文，防止大量 web 内容污染主 Agent 上下文 |
| D-6 | 连续工具错误处理 | 计数限制（P016 修复）：连续错误超阈值时主动退出 Loop | 无限重试 | web_search GFW 阻断时触发 200 次无限循环，必须有熔断机制 |
| D-7 | 本地 LLM 适配 | 自动发现（探测标准端口 11434/1234/8080/8000）+ tool-call 能力探测 + XML fallback | 手动配置 | 零配置体验；不支持 native tool-call 的模型自动降级 |

## 三、审查报告（高频 P0 问题）

> 注：Seed AI 经历 42 次迭代，问题是逐步暴露和修复的，非一次性审查。以下记录最具代表性的系统级问题。

### P0 级重复犯错（三次重犯同一问题）

| # | 问题描述 | 出现位置 | 修复方案 |
|---|---------|---------|---------|
| 1 | `compactWithSummary` 内直接 `new Anthropic()` | 压缩模块 | 改为 `createProvider(settings)` |
| 2 | 记忆提取（session end）直接 `new Anthropic()` | 记忆模块 | 改为传入 Provider 实例 |
| 3 | 科研数据源注入时直接 `new Anthropic()` | 系统提示模块 | 改为复用用户配置 Provider |

**根因**：每次新增"内部 LLM 调用"功能时，开发者（或 AI）习惯性用 `new Anthropic()` 作为快捷方式，而不是通过 Provider 接口。这是最典型的"接口抽象被绕过"模式。

### P0 级系统性问题

| # | 问题描述 | 位置 | 修复方案 |
|---|---------|------|---------|
| 1 | 连续工具错误触发无限循环（P016）：web_search GFW 阻断 → 200 次迭代耗尽上限 | Agent Loop | 增加连续错误计数器，超阈值主动退出 Loop |
| 2 | 工具层直接 throw Error，导致 Agent Loop 中断 | 多个工具 | 工具层 catch 所有异常，返回 `{ isError: true, content: "..." }` |
| 3 | 缓存 Key 仅含文件名，忽略行范围参数，导致脏读 | tool cache | Key = `toolName + JSON.stringify(全部入参)` |

## 七、经验教训

1. **Provider 接口抽象是第一优先级**：多 Provider 架构中，任何内部 LLM 调用都必须通过 Provider 接口；违反一次就会让某个 Provider 用户遭遇 403。这个原则需要在约束提示词第一条出现
2. **Agent Loop 必须有熔断机制**：连续工具错误不处理 = 无限循环；特别是网络类工具（web_search 被 GFW 阻断），必须有连续错误计数 + 阈值退出
3. **工具层永远不向上抛异常**：工具是 Agent Loop 的"外设"，抛异常会破坏整个 Loop；统一返回 `{ isError: true, content: "描述" }`
4. **缓存 Key 必须含全部入参**：文件读取缓存如果 Key 不含行范围，line 1-100 和 line 200-300 会命中同一个缓存条目，产生幽灵脏读
5. **非致命模块必须降级不崩溃**：记忆/沙箱/压缩等附加功能，失败时记录 warn 并继续启动，用户体验远好于启动崩溃
6. **输出截断是必须的基础设施**：一个无截断的 bash 工具理论上可以输出 GB 级数据，直接打爆上下文窗口；对所有工具输出设置硬上限
