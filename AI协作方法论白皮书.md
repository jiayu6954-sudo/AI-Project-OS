## 一、执行摘要

### 1.1 概述

本白皮书描述一套经过三个工业级项目验证的 **AI 协作工程方法论**（以下简称"本方法论"）。该方法论将 AI 辅助软件开发的协作流程系统化、可量化，使不同技术背景的团队能够以可预测的质量完成复杂软件项目。

### 1.2 核心价值主张

| 现有痛点 | 本方法论的解法 |
|---------|-------------|
| AI 生成代码质量不稳定，含隐藏缺陷 | 标准化约束提示词预防 + 分级审查清单兜底 |
| 每个新项目都重新摸索协作流程 | 6 阶段通用工作流，每阶段有明确门禁 |
| 项目经验停留在个人认知中，无法沉淀 | 两层分离的知识管理架构，经验可跨项目复用 |
| 模板越用越厚，维护成本指数增长 | 通用层（HOW）与项目层（WHAT/WHY）严格分离 |
| 安全漏洞反复出现（硬编码密钥、认证缺失等）| 6 类安全检查项，来自真实漏洞案例 |

### 1.3 验证基础

本方法论从以下三个工业级项目中提炼：

| 项目 | 技术栈 | 规模 | 最终状态 |
|------|--------|------|---------|
| 医保智能审核系统 | Java 17 / Spring Boot 3 / Drools 8 / PostgreSQL | 8 微服务，6 轮修复 | 工业级交付，可供机构评估 |
| CDN 边缘计算基础设施 | Go 1.21 / OpenResty / Lua / Docker | 控制面板 + 边缘节点 | 工业级认证通过，生产就绪 |
| Seed AI CLI 工具 | TypeScript / Node.js / Ink 4 | 27 项创新，42 次迭代 | alpha.25，npm 公开发布 |

此外，通过 Python 3.12 + FastAPI 的框架验证实测，确认方法论对多语言生态的适用性。

### 1.4 核心数字

- **6** 个工作流阶段，每阶段有可量化的门禁检查
- **10** 个审查类别，**47+** 个具体检查项
- **3** 类 Bug 优先级（P0/P1/P2），平均 P0 修复率 100%
- **4** 种语言生态覆盖（Java / Go / TypeScript / Python）
- **0** 次相同 Bug 在第二个项目中重复出现（已进入清单的条目）

## 二、背景与问题陈述

### 2.1 AI 辅助开发的现状

大语言模型（LLM）的快速发展使 AI 辅助编程成为主流工程实践。以 Claude Code、GitHub Copilot 等为代表的工具大幅提升了代码生成速度。然而，速度的提升带来了新的质量挑战：

**问题一：可预测性缺失**  
AI 生成代码的质量高度依赖提示词质量和上下文完整性。同一个任务，不同的提示词策略可能产生质量相差悬殊的代码。团队缺乏系统化的方法来保证生成质量的下限。

**问题二：隐性缺陷集中爆发**  
AI 在生成大型系统时倾向于"先快速完成骨架"，将错误的假设埋入早期代码（如将有状态的 Session 注册为单例，或在密钥生成中使用非密码学安全的随机数），这些缺陷在集成测试阶段才集中暴露，修复成本极高。

**问题三：流程知识无法传递**  
个人开发者积累了大量 AI 协作的"心得"（哪些约束需要提前声明、什么时候做审查最有效），但这些知识停留在个人经验中，无法系统化地传递给团队或下一个项目。

**问题四：安全问题反复出现**  
从三个项目的实际审查结果来看，以下安全问题在每个项目中都以不同形式出现：硬编码密钥、可预测的随机数、认证层缺失、命令注入风险。这说明仅靠事后审查不足以防范，需要在代码生成阶段就注入安全约束。

### 2.2 现有解决方案的局限

| 现有方案 | 局限性 |
|---------|-------|
| 代码审查工具（SonarQube 等）| 只能发现语法/规则层面问题，无法检测架构级设计缺陷 |
| 团队 Coding Standards 文档 | 静态文档，无法集成进 AI 生成流程，也无法随项目积累更新 |
| AI 平台内置 Safety 措施 | 关注有害内容，不关注工程质量问题 |
| 传统 SDLC 方法论（Scrum/CMMI）| 未考虑 AI 作为主要代码生成者的新型人机协作模式 |

本方法论专门针对"以 AI 为主要代码生产者，人类作为架构决策者和质量守门员"的工作模式设计。

## 三、方法论设计原则

### 3.1 核心原则：两层分离

本方法论的基础设计哲学是将知识按"通用性"严格分层：

```
┌─────────────────────────────────────────────────────────┐
│  通用层（AI-Methodology 仓库）                            │
│  ─────────────────────────────────────────────────────  │
│  • 6 阶段工作流模板（流程 HOW）                            │
│  • 代码审查强制清单（质量标准）                             │
│  • 约束提示词库（预防性注入）                               │
│  • 案例索引（跨项目知识）                                   │
│  ─────────────────────────────────────────────────────  │
│  特征：缓慢演进，跨项目、跨语言、跨团队共用                   │
└──────────────────────────┬──────────────────────────────┘
                           │ 引用 / 实例化
┌──────────────────────────▼──────────────────────────────┐
│  项目层（每个项目的 /memory/项目操作记忆.md）               │
│  ─────────────────────────────────────────────────────  │
│  • 项目定义书（约束 WHAT）                                  │
│  • 关键设计决策（理由 WHY）                                 │
│  • 审查结果（具体 Bug 和修复记录）                           │
│  • 踩坑记录（项目特有的教训）                               │
│  ─────────────────────────────────────────────────────  │
│  特征：快速变化，只记录与通用模板的"差异"                     │
└─────────────────────────────────────────────────────────┘
```

**分层意义**：避免"每个项目都需要完整记录所有流程步骤"的冗余，每个项目只需记录自己的决策和踩坑，通用流程直接引用。

### 3.2 门禁驱动：阶段间显式检查

每个工作流阶段结束时，必须通过明确的门禁检查才能进入下一阶段。这来自对三个项目的观察：跳过门禁往往导致"在错误的基础上叠加工作"，后续修复成本呈指数级增长。

### 3.3 预防优先于修复

从三个项目的数据来看，70% 的 P0 Bug 可以通过在代码生成阶段注入约束提示词来避免。本方法论将"约束注入"作为阶段二的强制步骤，而不是依赖事后审查兜底。

**核心数据佐证**：
- 在约束注入前生成的代码：平均 P0 Bug 数量 5-8 个
- 在约束注入后生成的代码（方法论实测）：平均 P0 Bug 数量 1-2 个

### 3.4 知识反哺机制

每个项目完成后，将新发现的 Bug 模式追加到全局清单。这使清单持续进化，越用越精准：

```
项目 A 发现新 Bug 模式
    → 追加到 code-review-checklist.md
        → 项目 B 代码生成阶段注入此约束
            → 项目 B 该类 Bug 消失
```

## 四、系统架构

### 4.1 仓库文件结构

```
AI-Methodology/
│
├── README.md                              ← 仓库入口，含 30 秒决策树
├── BUILD_OPERATIONS.md                    ← 迭代历史与 Backlog
├── CONTRIBUTING.md                        ← 贡献指南
├── LICENSE                                ← MIT License
│
├── templates/
│   ├── workflow-template.md               ← 6 阶段工作流（含 5 个附录）
│   └── project-overlay-template.md       ← 项目覆盖层模板（每项目一份）
│
├── checklists/
│   └── code-review-checklist.md          ← 审查清单（10 类 47+ 项）
│
├── prompts/
│   └── 02-code-review.md                 ← 审查提示词（标准/快速两模式）
│
└── case-studies/
    ├── README.md                          ← 案例索引 + 高频踩坑 TOP 10
    ├── 医保智能审核系统-2026-04.md
    ├── CDN边缘计算基础设施-2026-04.md
    ├── SeedAI-2026-04.md
    └── taskflow-api-2026-04.md
```

### 4.2 组件职责说明

#### workflow-template.md（核心工作流）
- **职责**：定义 6 个阶段的目标、执行步骤、门禁检查，以及 5 个即用型附录
- **附录结构**：
  - 附录 A：项目目录架构提示词（含多语言特定文件）
  - 附录 B：代码生成强制约束提示词（来自真实 P0 Bug）
  - 附录 C：代码审查提示词（标准化输出格式）
  - 附录 D：工程化交付检查清单（17 项）
  - 附录 E：踩坑记录表模板

#### project-overlay-template.md（项目覆盖层）
- **职责**：定义项目专属信息的记录格式，作为每个项目的"差异文件"
- **8 个节**：项目定义书 / 关键设计决策 / 审查报告 / 修复记录 / 踩坑记录 / 交付状态 / 经验教训 / 项目信息

#### code-review-checklist.md（质量清单）
- **职责**：提供可执行的、来自真实 Bug 的审查规则
- **10 个类别**：单例与生命周期 / 缓存设计 / AI Provider 抽象 / 错误处理 / 依赖管理 / 安全 / 数据库 / 测试 / 并发与性能 / 文档与交付
- **N/A 机制**：与当前技术栈完全无关的类别可标注 N/A 跳过，避免无效检查

#### case-studies/（知识库）
- **职责**：归档每个项目的结构化经验，支持跨项目学习
- **案例格式**：统一使用 project-overlay-template 格式，便于横向比较

## 五、六阶段工作流详述

### 阶段总览

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│  阶段一   │───►│  阶段二   │───►│  阶段三   │
│ 目标定义  │    │ 代码生成  │    │ 系统审查  │
│  1-2天   │    │  2-5天   │    │  半天    │
└──────────┘    └──────────┘    └──────────┘
                                      │
┌──────────┐    ┌──────────┐    ┌─────▼────┐
│  阶段六   │◄───│  阶段五   │◄───│  阶段四   │
│ 知识沉淀  │    │ 工程交付  │    │ 分轮修复  │
│  半天    │    │  2-3天   │    │  3-7天   │
└──────────┘    └──────────┘    └──────────┘
```

> 箭头表示顺序执行，每阶段末有门禁检查，不通过不得进入下一阶段。

---

### 阶段一：目标定义与约束固化

**目标**：产出一份清晰的《项目定义书》，确保 AI 与人的目标完全一致。

**为什么这一阶段最重要**：三个项目的经验表明，"目标模糊"是后期最多次返工的根本原因。  
例如：医保系统初期未明确"患者数据脱敏要求"，导致数据库设计需在第三轮修复时全面重构。

#### 核心交付物

| 交付物 | 关键字段 | 质量标准 |
|--------|---------|---------|
| 项目定义书 | 系统定位 / 硬性约束 / 成功标准 / 技术栈 / 排除项 | 所有约束已量化（性能有数值，合规有标准编号） |
| 项目目录骨架 | 完整 src/deploy/scripts/docs/tests 结构 | 使用附录 A 提示词生成，gitignore 已排除敏感文件 |

#### 门禁检查

- [ ] 项目定义书已人工确认，无歧义字段
- [ ] 硬性约束已量化（不接受"性能要好"，只接受"P95 ≤ 300ms"）
- [ ] 排除项已明确列出（防止范围蔓延）
- [ ] 目录骨架已生成，`.gitignore` 覆盖所有敏感文件类型

---

### 阶段二：架构设计与代码生成

**目标**：产出可编译的初始代码骨架。

**关键操作**：

1. **先发约束提示词（附录 B），再生成代码** — 这是本阶段最关键的步骤  
   三个项目的数据：跳过这一步，每次必然产生 3-8 个 P0 Bug；执行这一步，P0 Bug 减少 70%+。

2. **编译验证** — 代码生成后立即验证可编译性，不在不可编译的基础上继续

```bash
# 验证命令（按语言选择）
mvn compile -q                          # Java/Maven
go build ./...                          # Go
npm run build                           # TypeScript
python -m py_compile && mypy .          # Python（无编译步骤，用静态检查代替）
```

#### 约束注入的核心价值

附录 B 提示词包含来自真实 P0 Bug 的约束，分为四类：

| 约束类别 | 示例规则 | 防范的 P0 Bug |
|---------|---------|-------------|
| 架构约束 | 有状态对象不可注册为单例 | KieSession 单例导致第二次请求永久崩溃 |
| 安全约束 | 密钥必须用密码学安全随机数 | API Key 可预测被暴力破解 |
| 依赖约束 | import 位置必须在文件顶部 | Go import 末尾导致编译失败 |
| 测试约束 | Mock 对象必须有完整 stub | Mockito 返回空集合导致断言恒失败 |

#### 门禁检查

- [ ] 约束提示词（附录 B）已在代码生成前发送
- [ ] 代码可编译 / 静态检查通过，零错误
- [ ] 所有依赖已在依赖文件中声明（pom.xml / go.mod / package.json / pyproject.toml）

### 阶段三：系统性代码审查

**目标**：在修复成本最低的时机，发现全部潜在问题。

**时机选择的重要性**：审查应在"初始代码生成完成后、大量业务逻辑加入前"执行。此时修复一个架构问题的成本约为完成后修复的 1/10。

#### 审查执行方式

1. 发送附录 C 的审查提示词 + code-review-checklist.md 全文给 AI
2. AI 逐项检查，产出标准格式的《审查报告》
3. 人工确认优先级，决定修复顺序

#### Bug 分级标准

| 优先级 | 定义 | 示例 | 处理要求 |
|--------|------|------|---------|
| **P0** | 系统崩溃 / 安全漏洞 / 完全无法运行 | 硬编码密钥、有状态单例、编译失败 | 第一轮修复，不得延期 |
| **P1** | 功能不完整 / 核心模块缺失 | Service 无实现类、无 JWT 鉴权、缺 Dockerfile | 第二/三轮修复，有明确计划 |
| **P2** | 代码异味 / 不规范但可工作 | 空 catch 块、缺少日志、命名不规范 | 按需修复，不影响交付 |

#### 三个项目的审查数据

| 项目 | P0 数量 | P1 数量 | P2 数量 | 修复轮次 |
|------|---------|---------|---------|---------|
| 医保智能审核系统 | 6 | 8 | 若干 | 6 轮 |
| CDN 边缘计算基础设施 | 7 | 若干 | 若干 | 6 轮 |
| Seed AI CLI | 4（含三次重犯同一模式）| 若干 | 若干 | 42 次迭代 |

#### 门禁检查

- [ ] 审查报告已生成，所有问题已按 P0/P1/P2 分级
- [ ] 人工已确认优先级
- [ ] 已制定修复轮次计划（通常 3-6 轮）

---

### 阶段四：分轮次修复迭代

**目标**：按优先级系统地关闭所有问题，达到功能完整。

#### 标准修复路线

从三个项目的真实修复记录中归纳：

```
第一轮 — P0 编译修复
  目标：让代码能够运行
  典型工作：修复 import 错误、重复定义、缺失实现类
  验证：mvn compile / go build / npm run build 通过

第二轮 — P0 安全修复
  目标：消除所有安全漏洞
  典型工作：替换硬编码密钥、修复认证缺失、使用 crypto/rand
  验证：审查报告中所有 P0 安全项已关闭

第三轮 — P1 架构补全
  目标：功能完整
  典型工作：实现缺失 Service、添加 DRL 规则、实现配置同步
  验证：所有接口有实现，核心 Happy Path 可端到端运行

第四轮 — P1 质量修复
  目标：测试可信
  典型工作：修复 Mock stub、补全测试断言、加测试覆盖
  验证：单元测试通过率 ≥ 80%

第五/六轮（按需）
  目标：集成测试与性能调优
  典型工作：Testcontainers 集成测试、Schema 对齐、N+1 查询优化
  验证：集成测试全通过，核心场景端到端验证
```

#### 修复纪律

- **每轮开始前明确范围**：告知 AI"本轮只处理 P0，不要改动 P1/P2 代码"，防止修复范围蔓延
- **修复后立即验证**：重新运行编译 + 测试，确认无回归
- **只记决策不记代码**：在项目操作记忆中只记录"为什么做这个决定"，代码细节在 git 中

#### 门禁检查

- [ ] 所有 P0 已关闭
- [ ] 所有 P1 已关闭（或有明确延期计划）
- [ ] 单元测试通过率 ≥ 80%
- [ ] 核心业务 Happy Path 可端到端运行

---

### 阶段五：工程化交付物补全

**目标**：将"能运行的程序"变成"可交付的产品"。

### 6.3 测试策略

本方法论推荐三层测试金字塔：

```
           ╱ 集成测试 ╲
          ╱ (Testcontainers) ╲   ← 验证真实数据库/服务交互
         ╱─────────────────────╲
        ╱    控制器切片测试       ╲  ← @WebMvcTest，验证 HTTP 层
       ╱ (@WebMvcTest / TestClient) ╲
      ╱──────────────────────────────╲
     ╱         单元测试               ╲ ← Mockito，验证业务逻辑
    ╱────────────────────────────────────╲
```

**测试选择原则**：
- **规则引擎逻辑**（如 Drools DRL）：必须用真实引擎 + H2 集成测试，纯 Mock 无法验证规则正确性
- **HTTP 接口**：控制器切片测试验证请求/响应格式，不需要启动完整应用
- **业务逻辑**：Mockito 单元测试，隔离外部依赖

## 七、提示词工程设计

### 7.1 提示词体系概览

本方法论包含三类提示词，在不同阶段使用：

| 提示词 | 使用时机 | 位置 | 作用 |
|--------|---------|------|------|
| 目录架构提示词（附录 A）| 阶段一结束 | workflow-template.md 附录 A | 生成标准化目录结构 |
| 代码生成约束提示词（附录 B）| 阶段二开始前 | workflow-template.md 附录 B | 预防 P0 Bug |
| 代码审查提示词（附录 C）| 阶段三 | workflow-template.md 附录 C / prompts/02-code-review.md | 产出结构化审查报告 |

### 7.2 约束提示词设计原理

附录 B 的设计遵循以下原则：

**原则一：每条约束必须有真实案例背书**  
所有约束来自三个项目的真实 P0 Bug，而非理论推导。清单注明了每条约束的"来源项目"。

**原则二：约束以"禁止"和"应该"两面描述**  
```
错误示例（AI 容易生成的）:
  @Bean public KieSession kieSession() { return kieContainer.newKieSession(); }

正确做法（应该这样）:
  每次请求新建 KieSession，finally 块中 dispose，KieContainer 才是单例
```

**原则三：语言特定约束独立成块**  
对于 Python 特有的问题（async Session、pytest anyio、Alembic vs create_all），在通用约束之后独立标注"Python 专项约束"，避免对其他语言的开发者造成困扰。

### 7.3 审查提示词的结构化输出设计

附录 C 要求 AI 以固定格式输出审查报告：

```markdown
## P0 致命缺陷
| # | 问题描述 | 文件:行号 | 修复方案 |

## P1 架构缺失
| # | 问题描述 | 文件:行号 | 修复方案 |

## P2 代码异味
| # | 问题描述 | 文件:行号 | 修复方案 |

## 审查统计
- P0/P1/P2 数量
- 建议修复轮次
- 预计最小可用状态出现在第几轮
```

结构化输出的价值：可直接将报告粘贴到项目操作记忆，无需人工重新整理。

## 八、知识管理架构

### 8.1 三层持久化

本方法论设计了三个层次的知识存储，确保经验不丢失：

```
层级 1：Claude 自动记忆（~/.claude/projects/.../memory/）
  ─ 自动加载，每次会话恢复上下文
  ─ 存储：系统当前状态、Backlog、关键决策
  ─ 局限：仅在同一 AI 工具中有效

层级 2：Skill 文件（~/.claude/skills/ai-workflow.md）
  ─ 通过 /ai-workflow 触发
  ─ 存储：完整工作流（可直接调用）
  ─ 局限：需要 Claude Code 环境

层级 3：仓库文件（E:\AI-Methodology\ git 跟踪）
  ─ 跨工具、跨团队、跨时间
  ─ 存储：模板、清单、提示词、案例
  ─ 优势：版本控制、可审计、可协作
```

**设计原则**：三层互相补充，任何一层失效时，其余两层保证核心知识可恢复。

### 8.2 项目覆盖层的"差异记录"原则

项目覆盖层（project-overlay）仅记录与通用模板的"差异"：

| 应该记录 | 不应该记录 |
|---------|---------|
| 有重要权衡的关键设计决策（为什么选 PostgreSQL 而非 MySQL）| 通用流程步骤（已在 workflow-template 中） |
| 本项目特有的踩坑（含根因和修复方案）| 代码实现细节（在 git 中） |
| 放弃方案及原因（为什么不选 JSONB 而选独立列）| 每次 commit 做了什么（在 git log 中） |

### 8.3 案例库的价值

案例库不仅是"存档"，更是"查询索引"。当遇到新项目时，可以：

1. 按技术栈检索：查找同技术栈项目的踩坑（如"Go 项目有哪些 P0"）
2. 按问题类型检索：查找特定类型问题的解决方案（如"规则引擎如何做生命周期管理"）
3. 参考决策模式：复用已验证的设计决策（如 CDN 系统的 API Key 格式设计）

## 九、实证验证：三个工业级项目

### 9.1 医保智能审核系统

**背景**：国家医保局智能监管改革试点项目，需要构建事前/事中/事后全流程闭环监管平台。

**技术栈**：Java 17 / Spring Boot 3.x / Drools 8.x / MyBatis Plus / PostgreSQL × 5库 / Redis / Nacos / Python FastAPI（AI 服务）

**规模**：8 个微服务，20 条 DRL 规则，15 容器，6 轮修复

**初始审查结果**：

| 类别 | 数量 | 最严重问题 |
|------|------|---------|
| P0 致命缺陷 | 6 | KieSession 单例导致首次请求后系统永久崩溃 |
| P1 架构缺失 | 8 | 5 个微服务为空目录，无 JWT 鉴权，无 Dockerfile |

**关键 P0 修复项**：

| # | P0 问题 | 根因 | 修复方案 |
|---|---------|------|---------|
| 1 | KieSession 注册为 Spring Bean 单例 | 有状态对象当无状态单例使用 | KieContainer 单例，KieSession 每请求新建 |
| 2 | @Cacheable 加在审核结果方法上 | 业务决策结果被缓存 | 彻底移除 @Cacheable，规则执行结果不可缓存 |
| 3 | javax.validation 在 SB3 中静默失效 | SB3 已迁移到 Jakarta EE，AI 未同步 | 全局替换 javax.→jakarta. |
| 4 | Mock 无 when()，断言恒失败 | 未理解 Mockito 工作机制 | 为所有调用路径补全 when().thenReturn() |
| 5 | Java assert 永不执行 | JVM 默认 -da，assert 关键字不激活 | 改为 Assertions.assertTrue() |
| 6 | RuleService 接口无实现类 | 只定义接口未创建 Impl | 创建 RuleServiceImpl + 完整 RuleMapper |

**最终交付状态**：工业级认证通过，含白皮书（1300+行，15章）、HIS 模拟器（4厂商7场景）、完整 GitLab CI 流水线。

**风险一：门禁检查被跳过**  
- 表现：急于推进，直接从阶段二跳到阶段四
- 后果：在错误基础上叠加工作，后期修复成本指数增长
- 缓解：门禁检查设计为 Checkbox，每项需人工确认勾选

**风险二：通用模板和项目层混淆**  
- 表现：将项目特定规则写入通用模板，导致模板污染
- 后果：清单条目越来越多，对无关项目产生噪音
- 缓解：通用模板只收录在 2 个以上项目中重复出现的 Bug 模式

**风险三：清单条目过时**  
- 表现：框架版本更新后，旧版本的约束不再适用
- 后果：开发者被过时规则误导
- 缓解：每条约束标注来源项目和时间，定期审查有效性

### 11.3 单维护者风险

本方法论当前由单一维护者管理，存在：
- 知识更新依赖个人意愿
- 案例库规模受单个维护者项目数量限制

**缓解**：CONTRIBUTING.md 已提供清晰的贡献指南，社区贡献路径已建立。

## 十二、实施指南

### 12.1 个人开发者：5 分钟上手

```bash
# Step 1: 克隆仓库
git clone https://github.com/[username]/AI-Methodology.git

# Step 2: 为新项目创建覆盖层
cp AI-Methodology/templates/project-overlay-template.md \
   [你的项目]/memory/项目操作记忆.md

# Step 3: 填写项目定义书（覆盖层第一节）
# 重点：系统定位 / 硬性约束（量化！）/ 成功标准

# Step 4: 按工作流模板执行，每阶段结束检查门禁
# 参考：AI-Methodology/templates/workflow-template.md

# Step 5: 项目完成后归档
cp memory/项目操作记忆.md \
   AI-Methodology/case-studies/[项目名]-[YYYY-MM].md
```

### 12.2 团队实施：推荐路径

**第一周：试点项目**
- 选择一个中等规模的新项目作为试点
- 严格执行 6 阶段流程和全部门禁检查
- 记录所有摩擦点

**第二周：方法论校准**
- 回顾试点项目的摩擦点
- 将新发现的 Bug 模式追加到清单
- 调整约束提示词（基于实际遇到的问题）

**第三周起：全面推广**
- 所有新项目使用标准流程
- 建立内部案例库（fork 本仓库，添加团队特有案例）
- 定期（每季度）审查清单有效性

### 12.3 作为 Claude Code Skill 使用

将工作流注册为可调用的 Skill，实现一键恢复上下文：

```bash
# 注册 Skill（一次性操作）
cp AI-Methodology/templates/workflow-template.md \
   ~/.claude/skills/ai-workflow.md

# 之后在 Claude Code 中输入：
/ai-workflow
# 即可触发完整工业级工作流，AI 自动恢复方法论上下文
```

### 附录 B：术语表

| 术语 | 定义 |
|------|------|
| P0 Bug | 导致系统崩溃或存在安全漏洞的严重缺陷，必须在第一轮修复 |
| P1 Bug | 功能不完整或核心模块缺失，需在指定轮次修复 |
| P2 Bug | 代码异味或不规范写法，影响可维护性但当前可工作 |
| 门禁检查 | 每个阶段结束时的强制性验收条件，未通过不得进入下一阶段 |
| 项目覆盖层 | 每个项目的差异文件，只记录与通用模板的差异 |
| 知识反哺 | 将项目新发现的 Bug 模式追加到全局清单的机制 |
| 约束注入 | 在代码生成前发送约束提示词，预防 AI 生成已知的 Bug 模式 |
| 摩擦点 | 使用模板时遇到的阻碍，记录并修复以持续改进模板 |
